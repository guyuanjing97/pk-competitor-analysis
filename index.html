1

总览

三家竞品不是在做同一种 PK，而是在解决“参与、组织、场景覆盖”三个不同问题。

核心结论：PK赛的价值不是让答题看起来像游戏，而是把题库练习转化为一套可组织、可触达、可激励、可复盘的培训活动。企学院应以知识巩固为首期边界，避免把业务 KPI 竞赛混入同一产品。

1.1 三类解决方案

魔学院强对战体验：解决学员“愿不愿意反复练”
酷学院标准活动运营：解决管理员“怎么组织一场竞赛”
云学堂/绚星多玩法游戏中心：解决“不同学习目标怎么覆盖”
业务指标 PK组织经营竞赛：解决“如何推动业务目标达成”
配置题库、规则、范围、时间
→
触达任务、通知、活动入口
→
对战匹配、答题、即时反馈
→
激励积分、榜单、奖励
→
复盘人员、题目、知识点数据

1.2 最核心的3项优势与3项不足

优

三项核心优势：分别解决参与、组织和场景覆盖

优势 01 · 魔学院

C端参与驱动力强

为什么核心：学员愿意进入、完成并再次挑战，是 PK 产生培训价值的前提。

证据：入口支持快速匹配、好友邀请和组队；对局包含倒计时、双方进度、逐题反馈、结算、战绩和排行榜，形成完整反馈节奏。

企学院启示：保留短路径开局、即时反馈和再次挑战，首屏先解决“想不想玩”。
优势 02 · 酷学院

B端活动运营闭环完整

为什么核心：企业 PK 不是单局游戏，管理员必须能稳定完成创建、发布、校验、运营和复盘。

证据：教程流程覆盖基础信息、题库、计分、奖励、分组、范围、发布前冲突校验和活动数据；报表可查看活动类型、题量、场次和参与时间。

企学院启示：沿用标准活动管理骨架，把规则校验和数据复盘做成发布闭环。
优势 03 · 云学堂/绚星

场景分层与能力覆盖广

为什么核心：知识练习、团队竞技和业务指标竞赛目标不同，拆分产品才能避免规则互相牵制。

证据：“头脑竞技场”按游戏设置、试题、参与者创建知识竞赛；平台运营另设业务指标 PK 活动，配置对象、周期和活动类型。

企学院启示：首期聚焦知识 PK，主持竞技和业务指标赛作为独立场景分期建设。
痛

三项核心不足：学习承接、配置认知和实时公平

痛点 01 · 共性缺口

学习闭环停在成绩与答案

为什么最痛：热闹不等于学习有效；没有薄弱知识诊断和补学承接，PK 容易成为一次性促活。

证据：魔学院结算与回顾聚焦胜负、正确/错误题数、用时、正确答案和我的答案；酷学院现有报表聚焦活动、题量、场次与参与。现有证据均未体现知识点诊断、课程推荐或培训任务回流。

企学院补足：打通错题、知识点、推荐课程、培训计划和员工学习档案。
痛点 02 · 酷学院为主

配置与规则理解成本高

为什么最痛：管理员若无法快速预测得分与胜负，容易误配、反复试错，也难向学员解释规则。

证据：酷学院把题库、答题时长、剩余时间奖励、胜负奖励、个人/团队榜和分组冲突集中在一条配置链路；魔学院还存在同步/轮流、跳过、超时等答题规则。

企学院补足：提供场景模板、推荐默认值、计分示例和发布摘要，高级规则折叠。
痛点 03 · 魔学院为主

实时状态与公平性成本高

为什么最痛：匹配等待、对手掉线、超时和团队贡献一旦处理不清，会直接影响对局完成率和申诉率。

证据：流程明确出现无人在线排队、好友接受后开局、等待对方答题、倒计时、跳过和中途退出确认；这些状态证明实时对局需要完整异常与公平规则。此项是基于流程的风险推断，不代表竞品已发生稳定性故障。

企学院补足：首期限制规模，明确重连、超时、缺席、平局及团队平均/累加规则。
证据口径：截图中可直接看到的字段与流程标为“直接证据”；“未体现”仅指当前资料没有展示，不能等同于竞品绝对不具备；实时稳定性与公平性属于基于流程复杂度的产品风险推断。

1.3 如何转化为企学院方案

酷学院的骨架复用标准活动的创建、范围、发布、状态与数据链路。
魔学院的体验吸收匹配、倒计时、双方进度、结算与再次挑战。
绚星的分层知识 PK、主持竞技、业务指标竞赛分别建设。
企学院的差异连接题库、培训计划、课程、排行榜和员工档案。

  1. 企业培训为什么需要 PK，而不是再做一套考试？

  3. 学习型 PK 与业务指标型 PK 的边界在哪里？
  4. 企学院一期需要建设什么，哪些玩法应后置？

预期产出

  1. 三家竞品的用户、场景、流程和功能框架。
  2. 配置、对战、激励和复盘的成熟做法与缺口。
  3. 企学院 PK赛的产品定位、P0 闭环和演进路径。
  4. 可继续转化为 PRD、交互原型和研发评估的依据。

魔学院｜魔 PK

核心价值：用强实时感和社交发起降低练题启动成本。

4.1 背景、用户与特点

核心用户企业学员、培训管理员、活动运营者商业形态企业培训平台中的游戏化练习能力
核心特点1v1与组队并存、学员自主发起、快速匹配、实时进度价值重心先让学员愿意进入一局，再用胜率、积分和排行驱动复玩
证据更新：魔学院 B 端配置流程已由用户提供的后台流程图直接确认，以下分析以真实后台页面为依据。

4.2 B端配置流程

PK列表统一查看和管理已创建的PK活动
→
基本信息填写活动基础字段、选择项与时间等信息
→
PK内容配置PK所使用的内容并查看内容列表
→
设置配置参与关系，并通过弹窗选择相关人员
→
预览发布汇总检查三步配置后完成发布
阶段页面作用交互特点
PK列表承载活动管理与新增入口列表页进入创建流程，符合后台活动管理心智
第一步：基本信息集中完成活动基础信息配置采用分步表单，包含下拉选择、日期选择等常规控件
第二步：PK内容确定本场PK使用的内容配置后可进入内容列表检查已选内容
第三步：设置完成参与关系等活动设置通过独立弹窗选择人员，减少主表单的信息密度
预览与发布发布前汇总检查活动配置将预览作为发布前确认环节，降低规则误配风险

路径关键步骤解决的需求

4.5 具体功能点：组队同步答题

用户需求把个人练题转化为团队协作和对抗，提高参与感。
交互设计双方阵营、头像、PK条、题号和倒计时持续强化竞技状态。
主要问题最快回答者代表全队会放大抢答，慢答成员容易成为旁观者。
企学院启示最快答题仅作为速度赛模板，同时提供个人累加和团队平均分。

酷学院｜PK赛

核心价值：把知识竞赛变成管理员可配置、可发布、可监控、可导出的标准活动。

5.1 背景、用户与特点

核心用户培训管理员、组织管理员、活动运营者、企业学员商业形态运营活动体系中的标准知识竞赛产品
核心特点多人/组队、题库计分、分组、范围、多排行榜、独立报表价值重心降低正式竞赛的组织成本，保证活动可控与可审计

5.2 B端管理员流程

活动列表筛选、状态、类型
→
基本信息名称、时间、部门、说明
→
题目计分题库、题量、分值、时长
→
范围奖励分组、次数、榜单、奖惩
→
发布报表冲突校验、明细、导出

值得借鉴的状态约束包括：活动开始后限制修改影响公平性的题目与规则；已开始活动不能直接删除，只能提前结束；同一成员出现在多个小组时发布前阻断。这些规则保障了竞赛结果的可信度。

5.3 C端学员流程

学员进入“我的 PK”后先看到胜率、场次、胜场和 PK 经验，再按多人对战或组队 PK 选择活动。对战页持续展示时间、双方得分与选择状态，结束后展示胜负、得分、积分和奖励，并能继续查看记录与排行榜。

5.4 功能框架

活动管理

  • 列表筛选
  • 多人/组队PK
  • 状态管理
  • 分享海报

三步创建

  • 基本信息
  • 题库与题量
  • 计分与时长
  • 范围与分组

学员参与

  • 我的PK
  • 随机匹配
  • 答题对战
  • 记录与排行

数据报表

  • 人员汇总
  • 活动明细
  • 对局明细
  • 字段与导出

5.5 具体功能点：计分、排行与奖励

用户需求既要判断单局胜负，也要长期运营个人和团队竞赛。
交互设计题量、单题分值、时长和剩余时间奖励形成可计算总分。
主要问题参数和榜单密集，管理员难以预测规则最终效果。
企学院启示提供默认公式、实时样例和发布摘要，隐藏非必要高级参数。

5.6 数据报表

酷学院可按人员查看参与时间、胜/平/负、胜率和 PK 经验，再下钻到活动与对局明细并导出。它满足“谁参与、与谁对战、结果如何”的审计需求，但题目难度、知识点薄弱项和课程承接仍可加强。

6

云学堂/绚星｜游戏中心与 PK活动

核心价值：按学习目标拆分玩法，不让一套 PK 规则承担所有业务场景。

6.1 产品分层

知识学习

游戏中心

智慧塔、头脑竞技场、趣味闯关，以题目、速度、连答和通关衡量学习结果。

业务经营

平台运营 PK活动

目标挑战、排名 PK、个人/部门/团队、晒单和指标进度，以业务目标完成率衡量结果。

关键判断：学习型 PK 和业务指标型 PK 的数据模型、作弊风险、运营责任完全不同。企学院首期只做知识型 PK，业务指标竞赛应作为独立产品演进。

6.2 C端使用路径

进入游戏中心发现当前可参与玩法
→
选择玩法智慧塔 / 竞技场 / 闯关
→
完成挑战答题、成长、实时排行
→
获得结果奖励、排行、错题训练

6.3 功能框架

智慧塔

  • 个人爬塔
  • 好友1v1
  • 排位赛
  • 错题训练

头脑竞技场

  • 多人实时答题
  • 固定/随机题
  • 速度与连答分
  • 实时榜与大屏

趣味闯关

  • 个人/团队
  • 关卡与故事
  • 通关阈值
  • 关卡奖励

指标PK

  • 目标挑战/排名
  • 个人/部门/团队
  • 指标与晒单
  • 排名奖励

6.4 具体功能点分析

功能解决需求体验评价企学院启示
头脑竞技场年会、面授、直播中大量学员同一时间竞答群体氛围强，但依赖主持运营、并发和统一开赛作为 P1 主持模式，不作为首期默认形态
业务指标 PK用销售额、拜访量等指标推动个人或团队目标目标挑战与排名 PK 边界清楚，但非知识学习模型保留扩展接口，后续独立建设组织竞赛

小鹅通企学院产品方案建议

以考练提效立项，以学习闭环形成差异化。

产品定位：知识 PK 活动。企业管理员基于题库创建、面向指定组织或培训项目发布，学员通过 1v1 或组队答题完成知识巩固，结果进入排行榜、错题复盘和学习档案。

8.1 B端建议：四步创建

基础信息名称、封面、时间、说明、入口、范围
→
玩法与题目1v1/组队、题库、题型、题量、计时
→
计分与激励胜负、团队规则、次数、榜单、积分
→
发布确认规则摘要、人数与冲突校验、C端预览

8.2 P0 核心闭环

模块一期能力企学院复用点

8.4 关键交互原则

默认可用、规则可解释

提供推荐模板与默认计分；配置时展示一局样例，发布页自动生成学员可读规则。

公平优先、状态可控

正确性高于速度；活动开始后锁定题目与计分，非核心展示信息仍可调整。

异常有兜底

明确匹配超时、成员缺席、掉线重连、题目失效、提前结束和奖励失败处理。

活动后必须承接学习

结算不仅展示胜负，还应引导错题复练、课程回访和薄弱知识点补学。

8.5 核心成功指标

管理员效率创建完成率、平均配置时长、模板使用率
参与与稳定性触达率、参与率、匹配成功率、完成率、掉线率
持续参与人均场次、二次参与率、分享率
学习效果正确率、错题复练率、关联课程回访率
团队公平成员答题覆盖率、贡献分布、申诉率
管理复盘报表查看率、明细下钻率、导出率

8.6 最终判断

最佳路径不是首期复制完整游戏中心，而是先交付“标准知识 PK 活动”：用简洁配置和可靠规则完成 1v1/组队、题库、计分、排行和复盘，再利用企学院已有课程、培训计划、组织与学习档案形成差异化。大屏竞技、闯关和业务指标赛应分阶段独立演进。
